# Java Trace Analyzer

A high-performance, client-side Single Page Application for analyzing large Java application server trace files (traces.xml) generated by IBM ODM Team Server.

## Features

- ✅ **Client-Side Processing**: Handles files up to 500MB entirely in the browser
- ✅ **Streaming XML Parser**: Uses saxes for memory-efficient parsing
- ✅ **Web Worker Architecture**: Off-main-thread processing keeps UI responsive
- ✅ **IndexedDB Storage**: Efficient local storage via Dexie.js
- ✅ **Synthetic Time Offsets**: Prevents visual overlap of SQL events in the same second
- ✅ **Transaction Span Derivation**: Automatically matches begin/commit/rollback events with LIFO algorithm
- ✅ **Smart Call Stack Analysis**:
  - Derives transaction span call stacks from begin/end events
  - Computes longest common suffix from bottom up
  - Strips redundant portions from child events in grouped view
- ✅ **Interactive Timeline**: Konva.js-based canvas visualization with pan, zoom, and double-click expansion
- ✅ **Advanced Filtering**: Filter by thread, duration, SQL mode, and method name
- ✅ **Test-First Development**: Comprehensive unit tests with 100% pass rate

## Tech Stack

- **Runtime**: Bun
- **Framework**: Svelte 5 + SvelteKit
- **Styling**: Tailwind CSS
- **Database**: Dexie.js (IndexedDB wrapper)
- **XML Parsing**: saxes (streaming parser)
- **Visualization**: Konva.js (HTML5 Canvas)
- **Icons**: Lucide Svelte
- **Testing**: Vitest + Testing Library

## Getting Started

### Prerequisites

- Bun runtime installed (`curl -fsSL https://bun.sh/install | bash`)

### Installation

```bash
cd trace-tool/app
bun install
```

### Development

```bash
bun run dev
```

Open [http://localhost:5173](http://localhost:5173) in your browser.

### Testing

```bash
# Run all tests
bun test

# Run tests in watch mode
bun test --watch

# Run tests with UI
bun test:ui
```

### Build

```bash
bun run build
```

### Preview Production Build

```bash
bun run preview
```

## Trace Format

The analyzer is designed for the IBM ODM Team Server SQL trace format:

```xml
<teamserver-sql-log startDate="..." file="...">
  <tx thread="..." event="beginTransaction|commitTransaction|rollbackTransaction" time="...">
    <callStack><![CDATA[...]]></callStack>
  </tx>
  
  <sql thread="..." mode="executeQuery|executeUpdate" executionTime="..." count="..." time="...">
    <statement><![CDATA[SELECT ...]]></statement>
    <parameter index="1"><![CDATA[Int: [123]]]></parameter>
    <callStack><![CDATA[...]]></callStack>
  </sql>
</teamserver-sql-log>
```

### Key Features

- **Transaction Lifecycle Events**: begin, commit, rollback
- **SQL Queries**: With execution time, row count, parameters, and call stacks
- **Synthetic Time Offsets**: Events in the same second are offset to prevent overlap
- **Derived Transaction Spans**: Automatically created from matched begin/end pairs

## Architecture Decisions

### Why Web Worker?

Large trace files (up to 500MB) would block the main thread during parsing. The Web Worker architecture keeps the UI responsive while processing data in the background.

### Why Synthetic Offsets?

The trace format only includes timestamps to the second. Multiple SQL queries in the same second would visually overlap. Synthetic millisecond offsets are calculated per-thread to create visual separation while maintaining chronological order.

### Why IndexedDB?

Parsing 500MB files generates hundreds of thousands of events. Storing them in memory would crash the browser. IndexedDB provides efficient storage and querying without memory constraints.

### Why Test-First?

Following the spec's requirement for Test-First Development ensures:
- Correct implementation of complex algorithms (offset calculation, span matching)
- Confidence in refactoring
- Documentation through tests
- Early detection of edge cases

## Performance Characteristics

- **Parsing**: ~10MB chunks, streaming to avoid memory spikes
- **Batch Size**: 1000 events per IndexedDB write
- **Worker Buffer**: Max 5000 events before forced flush
- **Indexes**: Compound index on [type+thread+startTime] for efficient filtering
